AI for DevTeam

A blueprint for an on-prem LLM platform that reads your code, opens branches & merge requests, runs tests, reviews MRs, turns tickets into solutions, reports on team performance, edits documents, and answers questions through a chat-terminal — all wired into your local GitLab + CI and exposed as services to your Rails 8 app.
On-prem · no cloud Apple Silicon Mac mini · 32 GB GitLab + GitLab CI Rails 8.1 · Ruby 3.4 Ollama / llama.cpp Updated 2026-06-06

0 · Executive summary

You already have the system of record (the Rails app: projects, tickets, sprints, pull requests with files_data/tests_data/coverage, comments, documents, team metrics). What you want to add is an AI layer that turns that data — plus the live code in GitLab and the signals from CI — into actions and insight.

The shape

A small local model server (Ollama) on the Mac mini, fronted by an AI service layer that owns prompts, context assembly, and JSON-validated outputs. Rails calls it; CI calls it; the chat window calls it.

The contract

Every service consumes a slice of one DevContext JSON tree and returns schema-validated JSON — never free text the app has to scrape. That makes the whole thing testable, including with Cucumber.

The reality

A 32 GB Mac mini runs 7B–14B coder models comfortably and a 32B model slowly. Plan for 1–2 concurrent requests, a job queue, and overnight batch for heavy reports.

Reading guide. §1–4 are foundations (hardware, models, architecture, the context tree). §5 is the service-by-service catalog — each with what it is → data in → JSON out. §6–7 cover CI wiring and realistic performance. §8–12 are two concrete build plans (Python-core and Ruby-native) with a story list, a test plan per story, and how Cucumber fits.

1 · Hardware & the model-size budget

On Apple Silicon the GPU shares unified memory with the CPU, so "VRAM" is just your RAM. On a 32 GB machine, after macOS (~5–7 GB) and your Rails/Postgres/Redis stack (~3–5 GB), you realistically have ~22–24 GB for the model weights plus the KV cache (which grows with context length). That is the single most important constraint.

What actually fits, at the common Q4_K_M quantization, with headroom for an 8k–16k context KV cache:

Rule of thumb: on 32 GB, treat 14B as your daily driver, keep a 7B for fast/interactive work (chat, commit messages), and reserve 32B for queued, quality-critical jobs (deep code review, architecture suggestions) where you can tolerate slower output. If you can buy the 64 GB Mac mini, do it — it unlocks 32B comfortably and 70B at a squeeze.

3 · Architecture

Three planes: Rails (UI + system of record + orchestration), the AI service layer (prompts, context assembly, schema validation, model routing), and the model runtime (Ollama). GitLab and GitLab CI sit on the side and are reached through a bot account. Everything stays on the LAN.

Why a separate AI service layer instead of calling Ollama from Rails directly? Because the value is not the model call — it's context assembly + a stable JSON contract + validation + routing. Keeping that in one place means the chat window, a CI job, and a Sidekiq batch all get identical, testable behaviour. In Stack B (§9) this "layer" is simply a set of Ruby service objects inside Rails; in Stack A (§8) it's a small FastAPI process.

Think of it as a serialized snapshot of "everything relevant right now," assembled on demand from your Rails models (+ GitLab + CI). How it maps to what you already have. project→Project, tickets→Ticket, pull_requests[].files→PullRequest#files_data, pull_requests[].tests→PullRequest#tests_data, coverage_percent is already a column, documents→Document, team[].metrics is computed from tickets/PRs. The only genuinely new pieces are rag (embeddings index) and the per-file diff (pull from GitLab on demand).

Pruning matters. A ContextAssembler builds the tree, then trims to a token budget per service: code review gets the PR diffs + conventions + RAG; the performance report gets team[].metrics + sprint but no source. This is the difference between "fits in 8k tokens and runs fast" and "overflows the context and is slow/garbage."

S1 · Repo Reader / Ingestor

What it is. The entry point. Given a GitLab project URL it clones/pulls the repo (read-only, via the bot token), walks the tree, and produces (a) a pruned file tree + key-file summaries for the context, and (b) embeddings of code/doc chunks stored in pgvector so every other service can do semantic retrieval (RAG). Re-runs incrementally on each push webhook.

Data needed

  • project.gitlab_url, branch, bot token
  • File contents + language detection
  • Optional: existing index (for incremental update)

How it runs

  • Clone → chunk (by symbol/function, ~400 tokens) → embed → upsert to pgvector
  • Summarize key files with the 7B model (one line each)
  • Emit a manifest the ContextAssembler caches

Output JSON

{
  "project_id": 42, "ref": "main", "commit": "a1b2c3d",
  "files_indexed": 318, "chunks_embedded": 2741, "skipped": 22,   // binaries/vendored
  "languages": { "csharp": 61, "ruby": 120, "gherkin": 40 },
  "key_files": [
    { "path": "src/PrintQueue.cs", "summary": "Core job queue; enqueue/dequeue + retry" }
  ],
  "tree": ["src/", "src/services/", "features/", "spec/"],
  "index_status": "ready", "took_ms": 8400, "incremental": true
}

S2 · Branch & Merge-Request generator

What it is. Turns a ticket into a working branch and an MR. Two modes: scaffold (create branch + a stub commit + open a draft MR with a checklist) and attempt (let the coder model write a first-pass change for small, well-specified tickets, then open the MR for human review). It never merges — humans + CI gate that.

Data needed

  • Ticket (title, description, acceptance criteria, type)
  • Project conventions + branch-naming rule
  • RAG chunks of the files likely to change
  • Bot token with write_repository

Guardrails

  • Branch name from a template: feature/t-{id}-{slug}
  • "attempt" mode capped to N files / M lines; else scaffold only
  • Always draft MR; CI must pass before "ready"

Output JSON

{
  "ticket_id": 1010, "mode": "attempt",
  "branch": { "name": "feature/t-1010-pdf-export", "from": "main", "created": true },
  "changes": [
    { "path": "src/AddPdfExportService.cs", "action": "create",
      "rationale": "new service implementing the export per AC",
      "diff": "@@ … +58 -0" }
  ],
  "merge_request": {
    "iid": 1010, "title": "Draft: T-1010 Add PDF export for print queue jobs",
    "url": "http://gitlab.local/devteam/print-tdi/-/merge_requests/1010",
    "draft": true, "checklist": ["add tests", "update docs", "coverage ≥ 80%"]
  },
  "confidence": 0.62, "needs_human": true,
  "warnings": ["touched 1 file outside the predicted set"]
}

S3 · Test runner orchestrator

What it is. Mostly not an LLM service — it triggers the real test suite (preferably by kicking a GitLab CI pipeline so tests run in the canonical environment) and normalizes the results into the same tests_data shape your PullRequest already uses. A small 7B pass turns raw failures into a human summary + likely cause.

Data needed

  • Branch / MR ref to test
  • project.ci.test_command
  • CI job artifacts: JUnit XML, coverage, Cucumber JSON

Why CI, not local exec

  • Reproducible env; no "works on my mini"
  • Keeps the Mac mini's RAM for the models
  • One source of truth for coverage

Output JSON

{
  "ref": "feature/t-1010-pdf-export", "pipeline_id": 88213, "status": "failed",
  "summary": { "total": 142, "passed": 139, "failed": 2, "skipped": 1 },
  "coverage_percent": 79.4,
  "tests": [
    { "name": "exports a valid PDF", "suite": "Cucumber", "status": "passed", "time_ms": 120 },
    { "name": "handles 50 jobs ≤ 2s", "suite": "Unit", "status": "failed",
      "message": "expected ≤2000ms, got 2640ms", "file": "spec/perf_spec.rb:42" }
  ],
  "ai_summary": "1 perf test over budget by 32%; likely the synchronous PDF render. Consider batching.",
  "likely_culprit_files": ["src/AddPdfExportService.cs"]
}

S4 · Code review generator

What it is. The flagship. Given an MR diff + conventions + RAG context, it produces structured findings (not prose) — each anchored to a file+line, with a severity, category, explanation, and a suggested patch. Rails renders them in your existing Code Review page; CI posts them as MR discussion notes (§6).

Data needed (context slice)

  • PR/MR: title, description, linked ticket + AC
  • Per-file unified diff + surrounding content
  • project.conventions.pr_rules + style guide
  • RAG chunks of impacted symbols (callers/callees)
  • Latest test + coverage result

Output discipline

  • Severity ∈ {blocker, major, minor, nit, praise}
  • Category ∈ {bug, security, perf, style, test, design}
  • Every finding has file+line; optional suggestion patch
  • A verdict the UI shows as a badge

Output JSON

{
  "mr_iid": 1010, "model": "qwen2.5-coder:14b", "reviewed_files": 3,
  "verdict": "needs_work",           // approve | needs_work | request_changes
  "summary": "Solid structure; one resource leak and a missing negative test.",
  "findings": [
    {
      "id": "f1", "file": "src/AddPdfExportService.cs", "line": 47,
      "severity": "blocker", "category": "bug",
      "title": "PdfWriter not disposed on exception path",
      "detail": "If Process() throws, the writer leaks a file handle…",
      "suggestion": "using var writer = new PdfWriter(stream);",
      "confidence": 0.9
    },
    {
      "id": "f2", "file": "features/pdf.feature", "line": null,
      "severity": "major", "category": "test",
      "title": "No scenario for invalid/empty queue",
      "detail": "AC lists error handling but only the happy path is covered.",
      "suggestion": "Scenario: Export with empty queue → friendly error",
      "confidence": 0.78
    }
  ],
  "by_severity": { "blocker": 1, "major": 1, "minor": 0, "nit": 2, "praise": 1 },
  "covers_acceptance_criteria": false
}
Quality technique: run the review twice (or with two models) and only surface findings that both passes agree on, or have a second model act as a skeptic that tries to refute each finding. This kills most "plausible but wrong" comments — the #1 failure mode of LLM review.

S5 · Ticket → solution suggester

What it is. Reads a ticket and the relevant code (via RAG) and proposes an implementation plan: which files to touch, the approach, risks, a test plan, and an estimate hint. This is what feeds S2 (branch/MR) and S12 (test gen).

Data needed

  • Ticket + acceptance criteria + labels
  • RAG chunks of related modules
  • Project conventions; similar past tickets/MRs

Output discipline

  • Ordered steps with target files
  • Explicit risks & open_questions
  • A test_plan S12 can consume

Output JSON

{
  "ticket_id": 1010, "feasibility": "clear",   // clear | needs_clarification | risky
  "approach": "Add an AddPdfExportService that streams queue jobs to a PdfWriter…",
  "steps": [
    { "n": 1, "file": "src/AddPdfExportService.cs", "do": "new service + Handle()" },
    { "n": 2, "file": "src/PrintQueue.cs", "do": "expose ReadAllJobs()" }
  ],
  "test_plan": ["Cucumber: happy path export", "Unit: 50 jobs ≤ 2s", "Cucumber: empty queue"],
  "risks": ["PDF render is CPU-bound; perf AC is tight"],
  "open_questions": ["A4 or Letter default?"],
  "estimate_hint_points": 5, "related_mrs": [803, 912]
}

S6 · Team & per-user performance reporter

What it is. Consumes team[].metrics + sprint history (computed deterministically by Rails — the LLM does not invent numbers) and produces (a) a narrative report, (b) per-user cards, and (c) chart specs S8 renders. This is what answers your questions: "who's the fastest deliverer? who estimates best? what's the sprint status?"

Critical design rule: all numbers are computed in Ruby/SQL and passed in; the model only ranks, explains, and writes prose. This makes the report trustworthy and testable (the metrics have unit tests; the prose is checked for not fabricating figures).

Data needed

  • Per-user rolling metrics (cycle time, points, PRs, review turnaround, estimate accuracy, bug rate, coverage added)
  • Sprint burndown + velocity history
  • Time window + comparison baseline

Answers it powers

  • fastest deliverer rank by cycle time + throughput
  • best estimator rank by estimate_accuracy
  • sprint status burndown + risk flag

S7 · Document rewriter

What it is. Takes an existing document (spec, README, risk doc) and rewrites it to a goal — "make it formal," "shorten by half," "restructure into sections," "translate to Hebrew," "fill gaps from the code." Returns structured blocks so the editor can diff/preview, plus a changelog of what it changed and why.

Data needed

  • Source document (markdown/HTML) + its kind
  • Instruction / target style + length
  • Optional RAG context (to ground facts in code)

Output discipline

  • Block list (heading/paragraph/list/code/chart-placeholder)
  • changes log for review
  • Never drops facts silently — flags removals

Output JSON

S8 · Chart & media generator

What it is. Produces chart specifications (data + type + labels) and diagram/source for documents — not pixel rasters from the LLM (models can't draw reliably). The LLM decides what chart fits and emits a spec; a deterministic renderer (Chartkick/Vega/Mermaid/SVG) turns it into the actual image. For "images," it generates Mermaid/Graphviz/SVG diagrams; for true raster art you'd add a local Stable Diffusion model (optional, see §13 perf note).

Data needed

  • The dataset (from S6 or a query)
  • Intent ("show velocity trend", "architecture of X")
  • Target medium (PNG/SVG/Mermaid)

Renderers (deterministic)

  • Charts → Chartkick / Vega-Lite spec
  • Diagrams → Mermaid / Graphviz → SVG
  • Raster (optional) → local SD via ComfyUI

Output JSON

{
  "artifacts": [
    { "ref": "chart:export_perf", "renderer": "vega-lite", "type": "line",
      "title": "Export time vs job count",
      "data": [{ "jobs": 10, "ms": 480 }, { "jobs": 50, "ms": 2640 }],
      "encoding": { "x": "jobs", "y": "ms" },
      "alt_text": "Export time grows super-linearly past 30 jobs" },
    { "ref": "diagram:flow", "renderer": "mermaid",
      "source": "flowchart LR; Queue-->Service-->PdfWriter-->File" }
  ],
  "insert_into_document": 5
}

S9 · Chat terminal (the agent)

What it is. The chat window in your app, but it behaves like a terminal client: the user types natural language or slash-commands, and an agent loop decides which of S1–S18 (its "tools") to call, runs them, streams results, and can chain them. "Review MR 1010", "create a branch for T-1042 and draft the MR", "run the tests on my branch", "who's the fastest deliverer this sprint?", "rewrite the spec doc in Hebrew and add a burndown chart."

Tool-call + response JSON (one turn)

{
  "turn_id": "t_8821", "user": "review MR 1010",
  "plan": [{ "tool": "S4.code_review", "args": { "mr_iid": 1010 } }],
  "tool_results": [{ "tool": "S4.code_review", "ok": true, "ref": "review:1010" }],
  "reply_markdown": "Reviewed MR !1010 — verdict **needs_work**: 1 blocker (writer leak), 1 missing test. [Open review]",
  "citations": ["src/AddPdfExportService.cs:47"],
  "actions_taken": [], "confirm_required": false
}
Safety: any tool that writes (S2 branch/MR, S7 doc save) returns confirm_required:true and shows a diff before acting, unless the user pre-authorized in settings. Read-only tools (review, report, search) run freely.

6 · Connecting to CI/CD (GitLab)

There are two integration patterns; use both. A bot account (a dedicated GitLab user) with a project access token is the identity for all writes (branches, MR notes, approvals).

Pattern A — Webhook-driven (event → Rails → AI → GitLab)

GitLab fires a webhook on MR open/update, push, or pipeline finish. Rails receives it, verifies the secret token, enqueues a Sidekiq job, the AI service runs, and the result is posted back to GitLab via its REST API and stored in your app.

# config/routes.rb
post "/webhooks/gitlab", to: "gitlab_webhooks#receive"

# app/controllers/gitlab_webhooks_controller.rb
class GitlabWebhooksController < ActionController::API
  def receive
    head(:unauthorized) and return unless valid_token?            # X-Gitlab-Token
    case request.headers["X-Gitlab-Event"]
    when "Merge Request Hook" then Ai::ReviewMergeRequestJob.perform_later(params[:object_attributes][:iid])
    when "Pipeline Hook"      then Ai::SummarizePipelineJob.perform_later(params[:object_attributes][:id])
    end
    head :ok
  end
end

Pattern B — Pipeline-job-driven (CI calls the AI)

A stage in .gitlab-ci.yml calls the AI service over the LAN and posts the result. Good for gating ("fail the pipeline if a blocker is found") and for keeping the review in the pipeline log.

# .gitlab-ci.yml
ai_review:
  stage: review
  rules: [{ if: '$CI_PIPELINE_SOURCE == "merge_request_event"' }]
  script:
    - # call the on-prem AI service; it reads the MR + posts notes back
    - |
      curl -fsS -X POST http://mac-mini.local:8800/services/code_review \
        -H "Authorization: Bearer $AI_TOKEN" \
        -d "{\"project_id\":$CI_PROJECT_ID,\"mr_iid\":$CI_MERGE_REQUEST_IID}" \
        -o review.json
    - ruby ci/post_review.rb review.json     # post discussions; exit 1 on blocker
  allow_failure: true     # start advisory; flip to blocking once trusted

Creating a branch from a ticket

User clicks "AI: start work" on a ticket → Rails calls S2 → bot creates feature/t-{id}-… via the Branches API → opens a draft MR → links it back to the ticket (your PullRequest row).

Triggering tests

S3 calls the Pipelines API (POST /projects/:id/pipeline?ref=…); the pipeline-finish webhook returns results → normalized into tests_data + coverage.

Posting a review

Each finding becomes a positioned MR discussion (Discussions API) anchored to file+line; the verdict drives approve/unapprove.

Generation speed is dominated by memory bandwidth, then model size. Below are realistic ranges for Apple Silicon Mac mini (M-series, Metal, Q4_K_M). An M5-class chip with higher bandwidth lands at the upper end or above; treat these as conservative.

Concurrency

Plan for 1 heavy + 1 light request at a time. Ollama can hold ~2 models hot in 32 GB; a third forces an unload/reload stall. Use the Sidekiq queue to serialize heavy jobs.

KV cache & context

Long contexts eat RAM and slow prefill. Cap most services at 8k–16k tokens via pruning + RAG rather than stuffing whole files.

Batch the heavy stuff

Run nightly: full re-index, performance reports, flaky-test scan, release notes. Interactive hours stay free for chat + review.

Bottom line for 32 GB: excellent for chat, search, ticket analysis, test/commit generation, and standard MR review on the 14B model. The 32B deep-review is usable but belongs in a queue. If AI review becomes a blocking CI gate for a busy team, that's the moment to move the model runtime to a 64 GB Mac mini / Mac Studio (or add a second mini as a dedicated inference node) — the architecture doesn't change, only the OLLAMA_HOST.

11 · Testing strategy & using Cucumber

The hard part of testing AI features is that model output is non-deterministic. The strategy is to push determinism to the edges: validate structure always, record/replay model calls in CI, and reserve live-model checks for a separate nightly "eval" suite.

Five techniques that make LLM features testable

  1. Schema validation always. Every service output is checked against its JSON schema (§5). A malformed response is a test failure, independent of content. This alone catches most regressions.
  2. Record & replay (VCR-style). Capture real model responses once, replay them in CI so tests are deterministic and offline. Re-record intentionally when prompts change.
  3. Property/invariant assertions. E.g. S6: "every number in the narrative appears in the input metrics"; S4: "every finding has a file that exists in the diff"; S7: "no source fact dropped without a flag." These hold regardless of wording.
  4. Golden snapshots with tolerance. For stable prompts at temperature 0, snapshot the structure (keys, counts, severities) — not exact prose.
  5. LLM-as-judge eval (nightly, separate suite). Run the live model on a labelled set; a judge model (or rules) scores precision/recall of review findings, estimation MAE, etc. Track as a trend, gate releases on regressions — never block PR CI on it.

Can we use Cucumber? — Yes, and it fits perfectly

You already run Cucumber and have a Gherkin editor. Cucumber is ideal for the user-facing flows of these services because the features double as living documentation. Stub the model in the test env (return a recorded fixture) so scenarios are deterministic; keep the live-model checks in the nightly eval suite.

# features/ai_code_review.feature
Feature: AI code review on a merge request
  Background:
    Given the local AI model is stubbed to return a recorded review
    And a project "Print TDI" with an open MR !1010 touching "src/AddPdfExportService.cs"

  Scenario: Reviewer sees structured findings and a verdict
    When I request an AI review of MR !1010
    Then I should see a verdict badge "Needs work"
    And I should see a "blocker" finding on "src/AddPdfExportService.cs" line 47
    And each finding should have a severity and a category

  Scenario: A blocker fails the CI gate
    Given the AI review contains a "blocker" finding
    When the CI "ai_review" job runs
    Then the job exits non-zero
    And the finding is posted as an MR discussion on line 47
# features/ai_chat_terminal.feature
Feature: Chat terminal drives the AI services
  Scenario: Ask for the fastest deliverer
    Given seeded metrics where "Oren" has the lowest cycle time
    When I type "who is the fastest deliverer this sprint?" in the chat
    Then the reply should name "Oren"
    And the reply should cite his average cycle time

  Scenario: Creating a branch asks for confirmation
    When I type "/branch T-1042"
    Then I should be asked to confirm before any write
    And no branch is created until I confirm
Net: RSpec for units/contracts (fast, deterministic), Cucumber for the human-visible flows (review, chat, branch-from-ticket, doc rewrite, "who's fastest") with the model stubbed, and a separate nightly eval suite on the live model for quality trends. The same .feature files you write here can themselves be generated by S12 — the loop closes.
