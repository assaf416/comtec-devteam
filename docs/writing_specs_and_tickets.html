Writing Specs & Tickets

An Agile guide to creating clear, actionable, and well-structured tickets — tailored to the DevTeam Hub ticket model.

🎯 Why Good Tickets Matter

A well-written ticket is the contract between the product owner and the development team. It answers three questions:

  1. What needs to be done?
  2. Why does it matter?
  3. How will we know it's done?

Poorly written tickets cause context switching, rework, scope creep, and frustration. Good tickets are the single best investment you can make in team velocity.

🏷️ Ticket Kinds

Story

A user-facing feature or capability. The most common ticket type. Always written from the user's perspective.

As a [type of user], I want [goal], so that [benefit].

Meta Story (Epic)

A large body of work that is broken down into multiple stories. Meta stories are never assigned directly — they track progress across child tickets.

Bug Fix

A defect in existing functionality. Must include how_to_reproduce steps. Priority should reflect user impact, not developer effort.

Hotfix

A critical production bug requiring immediate attention. Bypasses normal sprint flow. Branch pattern: hotfix/T-<id>-<slug>.

Spike

A time-boxed research task. The deliverable is knowledge, not code. Always set a time limit (e.g., 4h). The outcome should be documented in the ticket description.

💡 Rule of thumb: If you can't demo it, it's a spike. If you can demo it, it's a story.

📝 Writing User Stories

The Format

As a [persona],
I want [action/feature],
so that [value/outcome].

The INVEST Criteria

Every good user story is:

LetterCriteriaMeaning
IIndependentCan be developed and delivered without depending on other stories.
NNegotiableDetails can be discussed — it's not a rigid contract.
VValuableDelivers something meaningful to the user or business.
EEstimableThe team can reasonably estimate the effort.
SSmallCan be completed in one sprint (ideally 1–3 days of work).
TTestableHas clear criteria to verify completion.

Story Splitting Strategies

If a story is too large (>8 story points), split it using one of these approaches:

  • By workflow step — Create, Read, Update, Delete as separate stories.
  • By business rule — Happy path first, then edge cases.
  • By data variation — Support one format first, then add others.
  • By interface — API first, then UI.
  • By operation — Simplest version first, then enhancements.

✅ Writing Acceptance Criteria

Acceptance criteria go in the description field. They define when the story is done.

Given-When-Then Format

Given [precondition],
When  [action],
Then  [expected result].

Checklist Format

For simpler stories, a checklist is acceptable:

Acceptance Criteria:
- [ ] User can see the login form on /login
- [ ] Email and password fields are validated
- [ ] Invalid credentials show an error message
- [ ] Successful login redirects to /dashboard
- [ ] Session persists across page reloads
💡 Tip: Acceptance criteria should be verifiable by QA without asking the developer. If a tester can't verify it from the description alone, it's not specific enough.

📚 Examples Using Our Ticket Model

Example 1: Story

🎫 T-42: Add password reset flow to login page

Kind:story
Level:moderate
Priority:high
Story Points:5
Dev Estimate:6h
QA Estimate:2h
Sprint:Sprint 12
Assignee:Jane Smith
Owner:Product Manager
Tags:auth, frontend
Description:
As a registered user,
I want to reset my password via email,
so that I can regain access to my account when I forget my password.

## Acceptance Criteria

Given I am on the login page,
When I click "Forgot password?",
Then I see a form asking for my email address.

Given I enter a valid registered email,
When I submit the form,
Then I receive a password reset email within 2 minutes.

Given I click the reset link in the email,
When I enter a new password (min 8 chars),
Then my password is updated and I am redirected to login.

Given I enter an unregistered email,
When I submit the form,
Then I see a generic "If this email exists..." message (no info leakage).

## Technical Notes
- Use Devise's recoverable module
- Reset token expires after 2 hours
- Rate-limit to 3 requests per email per hour
Test Plan:
1. Verify reset link appears on login page
2. Submit valid email → check email delivery
3. Submit invalid email → verify generic message (no enumeration)
4. Click reset link → change password → login with new password
5. Click expired link → verify error message
6. Try resetting 4 times → verify rate limit response

Example 2: Bug Fix

🐛 T-67: Fix crash when uploading CSV with Unicode characters

Kind:bug_fix
Level:simple
Priority:critical
Story Points:2
Dev Estimate:2h
QA Estimate:1h
Tags:import, encoding, backend
Description:
## Problem
Uploading a CSV file containing Hebrew or emoji characters causes a 500 error.
The import page crashes with an Encoding::InvalidByteSequenceError.

## Expected Behavior
CSV files with any valid UTF-8 content should import successfully.

## Impact
Blocks 3 customers who use Hebrew names in their data.
How to Reproduce:
1. Go to /imports/new
2. Upload the attached file: test_hebrew.csv
3. Click "Import"
4. Observe: 500 error page
5. Expected: Import succeeds, 12 records created
Test Plan:
1. Upload CSV with ASCII only → passes
2. Upload CSV with Hebrew characters → passes
3. Upload CSV with emoji → passes
4. Upload CSV with mixed encodings (Windows-1255) → graceful error message
5. Verify existing import specs still pass

Example 3: Spike

🔬 T-88: Evaluate Redis vs Memcached for session storage

Kind:spike
Level:moderate
Priority:medium
Story Points:3
Dev Estimate:4h (time-boxed)
QA Estimate:0h
Tags:infrastructure, performance
Description:
## Goal
Determine whether Redis or Memcached is better suited for our session
storage needs. Time-boxed to 4 hours.

## Questions to Answer
1. Which has lower latency for our session read pattern?
2. Which is easier to deploy with our current Docker setup?
3. What is the memory footprint for ~500 concurrent sessions?
4. Does Redis persistence add meaningful value for sessions?

## Deliverable
A summary in the ticket description with a recommendation and rationale.
No code changes expected.

Example 4: Hotfix

🚨 T-91: Hotfix — Payment webhook returning 404 in production

Kind:hotfix
Level:simple
Priority:critical
Story Points:1
Dev Estimate:1h
Tags:payments, production
Description:
## Incident
Production payment webhooks from Stripe are returning 404 since deploy v2.14.
Revenue impact: all new subscriptions are failing.

## Root Cause (suspected)
Route was renamed from /webhooks/stripe to /api/webhooks/stripe but
Stripe dashboard was not updated.

## Fix
Update routes.rb to accept both old and new paths.
Update Stripe dashboard webhook URL after deploy.
How to Reproduce:
1. Send a test webhook from Stripe dashboard to /webhooks/stripe
2. Observe: 404 response
3. Send to /api/webhooks/stripe → 200 OK

⭐ Best Practices

Writing Titles

✅ Good titles:
  • Add password reset flow to login page
  • Fix CSV import crash with Unicode characters
  • Show sprint velocity chart on dashboard
  • Evaluate Redis vs Memcached for sessions
❌ Bad titles:
  • Login page
  • Bug fix
  • Improve performance
  • Update stuff

Description Guidelines

  1. Start with context — Why are we doing this? What's the user problem?
  2. Be specific — "Users report slow load times" → "Dashboard takes 8s to load with >100 tickets"
  3. Include acceptance criteria — Every story needs them. No exceptions.
  4. Add technical notes — Architecture decisions, API contracts, database changes.
  5. Attach mockups — A screenshot or wireframe is worth a thousand words.

Workflow Best Practices

StatusWho moves itWhen
backlogProduct OwnerWhen the ticket is created but not yet prioritized
openProduct OwnerAfter sprint planning — ready to be picked up
in_progressDeveloperWhen work begins (dt ticket checkout does this automatically)
in_reviewDeveloperWhen PR is submitted (dt ticket commit does this)
testingReviewerAfter code review passes, QA begins testing
doneQA / DeveloperWhen all acceptance criteria pass
blockedAnyoneWhen external dependency prevents progress

Definition of Done (DoD)

A ticket is done when:

  • All acceptance criteria are met
  • Code has been reviewed and approved
  • Automated tests are written and passing
  • No regressions in existing test suite
  • actual_hours is filled in
  • Documentation is updated (if applicable)

Definition of Ready (DoR)

A ticket is ready for sprint when:

  • Title is clear and actionable
  • Description includes acceptance criteria
  • Story points are estimated
  • Dependencies are identified and resolved
  • Mockups/designs are attached (for UI work)
  • Test plan is drafted

🚫 Common Anti-Patterns

❌ The Vague Ticket

"Make the app faster" — No acceptance criteria, no definition of done, no way to know when you're finished.

✅ Instead

"Reduce dashboard load time from 8s to under 2s for accounts with 100+ tickets. Measure with PageSpeed Insights."

❌ The Kitchen Sink

"Build the entire reporting module" — Too large to estimate, too complex for one sprint, impossible to review.

✅ Instead

Create a meta_story and split into: "Show ticket count by status chart", "Add CSV export for sprint report", etc.

❌ The Solution Disguised as a Story

"Add a Redis cache to the User model" — Describes implementation, not the user problem.

✅ Instead

"As a user, I want the profile page to load in under 1 second." Let the developer choose the implementation.

❌ The Missing Bug Report

"Login doesn't work" — No reproduction steps, no expected vs actual behavior, no environment info.

✅ Instead

Fill in how_to_reproduce with numbered steps, browser/OS, and attach a screenshot.
