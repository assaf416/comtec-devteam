Why We Need Automatic Testing

Business reason: lower release risk, faster delivery, fewer regressions, better observability.

Current operational risks without broad automation

Manual checks scale poorly as project count grows. Defect discovery shifts late in the cycle, and environment-specific issues (especially in on-prem integrations) are detected too late.

Late bug discovery multiplies cost. In practical delivery economics, fixing a bug in production can cost an order of magnitude more than catching it in PR/CI.

BDD First: Gherkin As Shared Contract

BDD aligns product intent with executable tests and creates traceability from story to production checks.

Core process

1) Write acceptance criteria in Gherkin.
2) Implement steps against API/service boundaries.
3) Reuse scenarios for regression gates in Jenkins.
4) Link failing scenario back to ticket/project dashboard.

UI Testing Recommendation: Playwright

Playwright is the strongest fit for stable browser automation in our setup.

How we will use Playwright

- Smoke suite on each deployment candidate.
- Full regression nightly and pre-release.
- Store traces/screenshots/videos as Jenkins artifacts.
- Link failing tests to notifications and report dashboards.

Project-by-Project Test Examples

BDD scenario samples and test approach for each active project.

1) Print Server / TDI (.NET + VB)

Focus: queue stability, timeout recovery, locale defaults, printer profile behavior.

Feature: Printer network fault recovery
  Scenario: Job fails cleanly after timeout retry budget
    Given a running print queue
    When printer connectivity is lost mid-transfer
    Then the system retries according to policy
    And the job ends in Failed state with reason code

2) TDI2 (ASP.NET Core + RabbitMQ)

Focus: message workflows, concurrency safety, dead-letter handling.

Feature: Dead-letter routing
  Scenario: Invalid payload is moved to DLQ
    Given queue orders.inbound receives malformed payload
    When consumer validation runs
    Then message is rejected
    And message appears in orders.dlq with diagnostics

3) Digital Internet Services (Vue + NestJS)

Focus: customer API contract, auth flows, UI journey from portal to support ticket.

Feature: Customer support submission
  Scenario: Portal user opens a support ticket
    Given authenticated customer "Acme"
    When they submit issue "Connection timeout"
    Then ticket is created
    And dashboard shows status Open

4) Work Management System (Next.js + Bun)

Focus: workflow state transitions, Kanban consistency, search/index correctness.

Feature: Kanban state transition
  Scenario: Move item from In Progress to Review
    Given task T-431 in In Progress
    When user drags task to Review
    Then task status becomes In Review
    And audit activity is recorded

5) DevTeam Hub (Rails)

Focus: reports, notifications, integrations, ticket-to-branch behavior.

Feature: Assignment creates branch and notification
  Scenario: Assign ticket to developer
    Given ticket #88 exists in project "TDI2"
    When assignee is set to "noam@devteam.local"
    Then a branch is generated for that ticket
    And assignee receives notification with branch command
