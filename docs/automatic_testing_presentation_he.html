למה אנחנו צריכים בדיקות אוטומטיות

הסיבה העסקית: פחות סיכון בשחרורים, מסירה מהירה יותר, פחות רגרסיות ויכולת תצפית טובה יותר.

הסיכונים התפעוליים הקיימים בלי אוטומציה רחבה

בדיקות ידניות לא גדלות טוב ככל שמספר הפרויקטים עולה. גילוי תקלות נדחה לשלבים מאוחרים במחזור, ובעיות תלויות סביבה, במיוחד באינטגרציות און-פרם, מתגלות מאוחר מדי.

גילוי מאוחר של באגים מגדיל עלות בצורה חדה. בפועל, תיקון תקלה בפרודקשן עלול לעלות סדר גודל יותר מאשר תפיסה שלה כבר ב-PR או ב-CI.

הכלים המתאימים ביותר לסטאק האון-פרם שלנו

הכלים שנבחרו נותנים עדיפות לאירוח עצמי, הרצה צפויה ואינטגרציה חזקה עם CI.
שכבה כלים מומלצים למה זה מתאים לאון-פרם איך נשתמש בזה
BDD Cucumber (Ruby/JS), SpecFlow (.NET), מדריך סגנון משותף ל-Gherkin בלי תלות ב-SaaS, קוד נקי + runner, משתלב בקלות ב-Jenkins קבצי Feature הופכים למקור האמת של קריטריוני הקבלה.
Unit + Service/API RSpec (Rails), bun:test (Bun), xUnit/NUnit (.NET) הרצה מהירה מקומית וב-CI, אקוסיסטמים יציבים כל merge request מריץ חבילות בדיקה דטרמיניסטיות לפני אינטגרציה.
UI E2E Playwright runners באירוח עצמי, תמיכה במטריצת דפדפנים וארטיפקטים של trace/screenshot מסלולים קריטיים, smoke pack לכל deployment, ורגרסיה לילית.
דיווח JUnit XML + Allure מבוסס ארטיפקטים, קל לפרסום מתוך Jenkins דשבורדים של pass/fail, בדיקות flaky ומגמות לפי suite או פרויקט.
ביצועים k6 + Grafana (self-hosted) רץ כולו בתוך הרשת הפנימית פרופילי עומס ומאמץ לפני שחרור ובדיקות baseline ליליות.

BDD קודם: Gherkin כחוזה משותף

BDD מיישר בין כוונת המוצר לבדיקות ביצועיות ויוצר עקיבות מסטורי ועד בדיקות פרודקשן.

התהליך המרכזי

1) כותבים קריטריוני קבלה ב-Gherkin.
2) מממשים צעדים מול גבולות API או שירות.
3) משתמשים מחדש בתרחישים כשערי רגרסיה ב-Jenkins.
4) מקשרים תרחיש שנכשל חזרה לטיקט ולדשבורד הפרויקט.

המלצה לבדיקות ממשק: Playwright

Playwright הוא ההתאמה החזקה ביותר לאוטומציית דפדפן יציבה בתצורה שלנו.

איך נשתמש ב-Playwright

- Smoke suite על כל מועמד ל-deployment.
- רגרסיה מלאה בלילה ולפני שחרור.
- שמירת traces, screenshots ו-videos כארטיפקטים של Jenkins.
- קישור בדיקות כושלות להתראות ולדשבורדי הדיווח. 

// playwright.config.ts (קונספט)
projects: [
  { name: 'chromium' },
  { name: 'firefox' }
],
reporter: [['junit', { outputFile: 'playwright-junit.xml' }]],
use: { trace: 'retain-on-failure', screenshot: 'only-on-failure' }

דוגמאות בדיקה לפי פרויקט

דוגמאות תרחישי BDD וגישת הבדיקה לכל פרויקט פעיל.

1) Print Server / TDI (.NET + VB)

פוקוס: יציבות תורים, התאוששות מ-timeout, ברירות מחדל של locale והתנהגות פרופילי מדפסת.

Feature: Printer network fault recovery
  Scenario: Job fails cleanly after timeout retry budget
    Given a running print queue
    When printer connectivity is lost mid-transfer
    Then the system retries according to policy
    And the job ends in Failed state with reason code

2) TDI2 (ASP.NET Core + RabbitMQ)

פוקוס: תהליכי הודעות, בטיחות קונקרנציה וטיפול ב-dead-letter.

Feature: Dead-letter routing
  Scenario: Invalid payload is moved to DLQ
    Given queue orders.inbound receives malformed payload
    When consumer validation runs
    Then message is rejected
    And message appears in orders.dlq with diagnostics

3) Digital Internet Services (Vue + NestJS)

פוקוס: חוזה API ללקוחות, תהליכי הזדהות וזרימת UI מהפורטל לטיקט תמיכה.

Feature: Customer support submission
  Scenario: Portal user opens a support ticket
    Given authenticated customer "Acme"
    When they submit issue "Connection timeout"
    Then ticket is created
    And dashboard shows status Open

4) Work Management System (Next.js + Bun)

פוקוס: מעברי סטטוס ב-workflow, עקביות Kanban ותקינות חיפוש ואינדוקס.

Feature: Kanban state transition
  Scenario: Move item from In Progress to Review
    Given task T-431 in In Progress
    When user drags task to Review
    Then task status becomes In Review
    And audit activity is recorded

5) DevTeam Hub (Rails)

פוקוס: דוחות, התראות, אינטגרציות והתנהגות ticket-to-branch.

Feature: Assignment creates branch and notification
  Scenario: Assign ticket to developer
    Given ticket #88 exists in project "TDI2"
    When assignee is set to "noam@devteam.local"
    Then a branch is generated for that ticket
    And assignee receives notification with branch command
