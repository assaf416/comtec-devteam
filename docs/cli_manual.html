dt CLI Manual

The DevTeam Hub terminal client. Manage tickets, run tests, deploy, and review code — without leaving your terminal.

v1.0.0 Ruby macOS / Linux

📦 Installation

The dt CLI is a single Ruby script with no gem dependencies beyond the standard library. Symlink it to your PATH:

# From the project root
$ ln -sf "$(pwd)/cli/dt" /usr/local/bin/dt

Verify it works:

$ dt --version
dt 1.0.0
💡 Tip: The CLI reuses the same ~/.devteam.yml config file as the devteam CLI. If you already ran devteam setup, you're ready to go.

🔧 First-Time Setup

Configure the CLI with your DevTeam Hub server URL and API token:

Terminal — dt setup
$ dt setup
──────────────────────────────────────────────────
  DevTeam Hub — Setup
──────────────────────────────────────────────────
  Hub URL [http://localhost:3000]: https://hub.mycompany.com
  API token (from https://hub.mycompany.com/api/v1/token): abcdef123456
  ✓ Authenticated as Jane Smith (developer)
  Config saved to /Users/jane/.devteam.yml

Your API token is available at {hub-url}/api/v1/token after logging in to the web interface.

Configuration is stored in ~/.devteam.yml:

# ~/.devteam.yml
host: https://hub.mycompany.com
token: abcdef123456

🎫 Ticket Management

All ticket commands follow the pattern: dt ticket <command> [args]. Shorthand: dt t <command>.

Most commands auto-detect the ticket ID from the current git branch (e.g. feature/T-42-login-page).

List tickets

# List your assigned tickets
$ dt ticket list

# List all tickets across projects
$ dt ticket list --all

# Filter by status
$ dt ticket list --status in_progress

# Filter by project
$ dt ticket list --project 3
Terminal — dt ticket list
$ dt ticket list

Show ticket details

$ dt ticket show 42

# Or auto-detect from current branch
$ dt ticket show

Change ticket status

# Reopen a ticket
$ dt ticket open 42

# Cancel a ticket (sets to closed)
$ dt ticket cancel 42

# Close / mark done
$ dt ticket close 42

Update ticket fields

Pass field-value pairs after the ticket ID:

$ dt ticket update 42 status done
$ dt ticket update 42 priority high
$ dt ticket update 42 status in_review priority critical
FieldAccepted Values
statusbacklog, open, in_progress, in_review, testing, done, closed, blocked
prioritylow, medium, high, critical
kindstory, meta_story, bug_fix, spike, hotfix
leveltrivial, simple, moderate, complex, expert
pointsInteger (story points)
titleAny string

Assign a ticket

# Assign to yourself
$ dt ticket assign 42

# Assign to a team member (name match)
$ dt ticket assign 42 to John

Move to a sprint

$ dt ticket move 42 to sprint 5

Checkout ticket branch

Creates or switches to the ticket's git branch and auto-sets status to in_progress:

$ dt ticket checkout 42

# Shorthand
$ dt ticket co 42
Terminal — dt ticket checkout
$ dt ticket checkout 42
──────────────────────────────────────────────────

💡 Branch naming: Branches are auto-named based on ticket kind: feature/T-<id>-<slug> for stories, bugfix/T-<id>-<slug> for bugs, hotfix/T-<id>-<slug> for hotfixes, spike/T-<id>-<slug> for spikes.

Run tests

Triggers a CI run for the ticket via the DevTeam Hub API (Jenkins integration):

# Run tests for a specific ticket
$ dt ticket run 42

# Auto-detect ticket from branch
$ dt ticket run

🔬 CI / Testing

Manage CI test runs directly from the terminal. All commands: dt ci <command>.

Run tests

# Run all tests
$ dt ci test

# Run a specific test scope
$ dt ci test frontend
$ dt ci test backend
$ dt ci test ui
ScopeDescription
allRun the full test suite (default)
frontendRun frontend / JavaScript tests only
backendRun Rails / backend tests only
uiRun UI / integration / Cucumber tests only

Rerun failing tests

Finds the most recent failed CI run for the current ticket and reruns it:

$ dt ci rerun

Cancel running tests

$ dt ci cancel

Check CI status

$ dt ci status

🚀 Deployment

Deploy the current branch to an environment. Shorthand: dt d.

# Deploy to staging
$ dt deploy staging

# "to" is optional syntactic sugar
$ dt deploy to staging

# Deploy to production (requires confirmation)
$ dt deploy production
Terminal — dt deploy production
$ dt deploy production
──────────────────────────────────────────────────
  Deploy to PRODUCTION
──────────────────────────────────────────────────
  Branch:  feature/T-42-login-page-redesign
  Ticket:  T-42

  Deploy to PRODUCTION? [y/N]: y

  Push branch...    ✓
  Run migrations... ✓
  Restart server... ✓
  ✓ T-42 marked done
  ✓ Deploy to production complete!
⚠ Production deploys require confirmation. When deploying to production, the linked ticket is automatically set to done.

The deploy process runs these steps:

  1. Push branch — git push -u origin <branch>
  2. Run migrations — bin/rails db:migrate RAILS_ENV=<env>
  3. Restart server — touch tmp/restart.txt

📜 Log

View git history or deployment logs. Shorthand: dt l.

# Recent git commits (last 15)
$ dt log

# Git log for a specific file
$ dt log app/models/ticket.rb

# Deployment history for staging
$ dt log staging

# Deployment history for production
$ dt log production
💡 Smart detection: When the argument is staging or production, dt log shows the deployment history from the DevTeam Hub API. Any other argument is treated as a file path for git log.

Branch auto-detection

When a ticket ID is omitted, dt extracts it from the current git branch name. Recognized patterns:

feature/T-42-login-page      → Ticket ID: 42
bugfix/T-18-fix-crash        → Ticket ID: 18
hotfix/T-99-urgent-patch     → Ticket ID: 99
chore/T-7-update-deps        → Ticket ID: 7
spike/T-33-evaluate-redis    → Ticket ID: 33

🔄 Typical Workflow

A complete ticket lifecycle from the terminal:

# 1. See your tickets
$ dt ticket list

# 2. Start working on a ticket
$ dt ticket checkout 42

# 3. ... write code ...

# 4. Run tests
$ dt ticket run

# 5. Check CI status
$ dt ci status

# 6. Commit and push
$ dt ticket commit

# 7. Review the diff
$ dt ticket review

# 8. Deploy to staging
$ dt deploy staging

# 9. Deploy to production
$ dt deploy production
