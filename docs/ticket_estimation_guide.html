Ticket Estimation

A practical guide to estimating work in Agile — using story points, complexity levels, and hour estimates from the DevTeam Hub model.

🧮 Estimation Fields in DevTeam Hub

Our ticket model provides multiple estimation dimensions. Each serves a different purpose:

FieldTypePurposeWho Sets It
story_points Integer Relative effort/complexity. Used for sprint planning and velocity tracking. Team (during planning)
level Enum Complexity classification: trivial, simple, moderate, complex, expert. Tech lead or developer
dev_estimate_hours Decimal Estimated developer hours for implementation. Assigned developer
tester_estimate_hours Decimal Estimated QA hours for testing. QA engineer
actual_hours String Time actually spent (e.g. "2d 4h"). Recorded after completion. Developer
actual_velocity Integer Actual story points delivered (may differ from estimate). Team (during retro)
estimated_by User ref Who last updated the estimate. Auto-set on ticket update. System (automatic)
💡 Key insight: Story points measure complexity, not time. A 5-point story for a senior developer and a junior developer takes different hours — but the complexity is the same.

What Each Point Value Means

PointsComplexityTypical WorkExample
1 Trivial Config change, copy update, one-line fix Change button label from "Submit" to "Save"
2 Simple Well-understood, small scope, clear path Add a new column to an existing table, add validation
3 Moderate Standard feature, some design decisions Add email notification when ticket is assigned
5 Significant Multiple components, moderate uncertainty Build CSV export with filters, pagination, and download
8 Complex Cross-cutting concern, integration work, new patterns Integrate third-party payment gateway with webhooks
13 Very complex Multiple unknowns, architectural changes, high risk Migrate session storage from cookies to Redis cluster
21 Epic-sized ⚠ Too large — must be split into smaller stories Build entire reporting module (split this!)
⚠ Rule: If a story is estimated at 13+ points, the team should discuss splitting it before committing it to a sprint. Stories above 8 points carry significant estimation risk.

⚙️ Complexity Levels

The level field classifies the technical complexity independently of effort. It helps with task assignment and load balancing.

LevelDescriptionWho Should Work On ItTypical Points
trivial No design decisions. Clear, mechanical change. Any team member, including juniors 1–2
simple Well-understood pattern. One component affected. Junior or mid-level developer 2–3
moderate Some design choices. Multiple components involved. Mid-level developer 3–5
complex Significant design decisions. New patterns or integrations. Senior developer 5–8
expert Architectural impact. Deep domain or infrastructure knowledge. Senior / tech lead with domain expertise 8–13

Level vs Points — They're Different

Level = Technical Difficulty

How hard is the problem? Does it require deep expertise? How many unknowns exist?

Points = Total Effort

How much work is involved? Includes implementation, testing, review, and deployment.

A simple level ticket can still be 5 points if it involves many repetitive changes across files. A complex ticket might be only 3 points if the solution is small but requires deep understanding.

🃏 Estimation Techniques

1. Planning Poker

The most common Agile estimation technique. Best for team-wide consensus.

  1. Product owner reads the ticket aloud
  2. Team discusses and asks clarifying questions
  3. Each member privately picks a Fibonacci card (1, 2, 3, 5, 8, 13, 21)
  4. All cards are revealed simultaneously
  5. If estimates differ by more than 2 steps, the highest and lowest explain their reasoning
  6. Re-vote until consensus (within 1 step)
💡 Time-box: Spend maximum 5 minutes per ticket. If you can't reach consensus, the ticket probably needs more refinement.

2. T-Shirt Sizing

Quick and informal. Good for backlog grooming when precise numbers aren't needed yet.

SizeEquivalent PointsLevel
XS1trivial
S2–3simple
M5moderate
L8complex
XL13+expert (needs splitting)

3. Reference Story Comparison

Compare each new ticket to a "reference story" the team has already completed. This anchors estimates in reality.

  1. Pick 3 completed tickets as reference points (small, medium, large)
  2. For each new ticket, ask: "Is this bigger or smaller than our reference?"
  3. Assign points relative to the references

4. Three-Point Estimation (for hours)

When hour accuracy matters (client billing, deadline forecasting), use three estimates:

Estimate = (Optimistic + 4 × Most Likely + Pessimistic) ÷ 6

Example: A feature might take 4h (best case), 8h (most likely), or 20h (worst case):

(4 + 4 × 8 + 20) ÷ 6 = 9.3h → set dev_estimate_hours to 9.5

📊 Tracking Actuals & Velocity

Why Track Actuals?

Comparing estimates to actuals over time reveals:

  • Estimation accuracy — Are we consistently over or under-estimating?
  • Team velocity — How many points can we reliably deliver per sprint?
  • Capacity planning — How much work can we commit to next sprint?
  • Process improvement — Which types of work have the widest estimate gaps?

Velocity Formula

Sprint Velocity = Sum of story_points for all tickets marked done in the sprint

Use the rolling average of the last 3 sprints for planning. Don't use a single sprint — it's too volatile.

Estimation Accuracy

Accuracy = dev_estimate_hours ÷ actual_hours × 100%

Good teams aim for 80–120% accuracy. Below 80% means systematic under-estimation. Above 120% means padding.

Using the CLI to Track

# Update actual hours after completing work
$ dt ticket update 42 actual_hours "1d 6h"

# View estimation accuracy in reports
# Reports → Estimation Accuracy (in the web UI)
💡 Best practice: Fill in actual_hours immediately when moving a ticket to done. Waiting until the end of the sprint leads to inaccurate numbers.

⭐ Estimation Best Practices

  1. Estimate as a team — The collective wisdom of the team is more accurate than any individual. Use Planning Poker.
  2. Estimate complexity, not duration — "How hard is this?" not "How long will it take you?"
  3. Include everything — Implementation + tests + code review + documentation. Not just the fun coding part.
  4. Use reference stories — Compare to past work the team has actually delivered.
  5. Don't estimate bugs at 0 — Even a one-line fix needs investigation, testing, and review.
  6. Time-box spikes — Always set a maximum time. The output is knowledge, not perfection.
  7. Re-estimate when scope changes — If requirements change mid-sprint, update the estimates.
  8. Track and learn — Review estimate accuracy in retrospectives. Adjust calibration over time.
  9. Don't game the numbers — Estimates are forecasts, not commitments. Padding destroys trust.
  10. Split, don't inflate — If a story feels too big, split it into smaller stories instead of giving it a huge number.

🚫 Common Estimation Pitfalls

❌ Anchoring Bias

The first person to say a number influences everyone else. That's why Planning Poker uses simultaneous reveal.

❌ Optimism Bias

"It's just a simple change" — until you discover the test suite, the edge cases, and the code review feedback.

❌ Confusing Effort with Duration

An 8-point story isn't "8 hours of work." It might be 16 hours spread across 3 days with interruptions.

❌ Not Including QA

A feature isn't done when the code is written. Include tester_estimate_hours in sprint capacity.

❌ Hero Estimation

"I can do this in 2 hours" — but what if you're sick? Estimate for the average team member, not the fastest.

❌ Ignoring Unknowns

If a ticket has research needed, add a spike first. Don't estimate implementation when you don't understand the problem.

The 90% Done Trap

⚠ Warning: "The first 90% of the work takes 90% of the time. The remaining 10% takes the other 90% of the time." — Tom Cargill

This usually happens when estimates don't include:

  • Error handling and edge cases
  • Cross-browser / cross-device testing
  • Code review iterations
  • Documentation updates
  • Migration and deployment considerations

📋 Quick Reference Card

Sprint Planning Checklist

  1. Review team velocity (average of last 3 sprints)
  2. Account for planned absences, holidays, meetings
  3. Pull top-priority tickets from backlog
  4. Estimate each ticket using Planning Poker
  5. Set story_points, level, dev_estimate_hours, tester_estimate_hours
  6. Assign tickets to team members based on level and expertise
  7. Verify total committed points ≤ team velocity
  8. Move tickets to sprint: dt ticket move <ID> to sprint <SID>

After Completing a Ticket

# 1. Record actual time spent
$ dt ticket update 42 actual_hours "1d 4h"

# 2. Mark as done
$ dt ticket close 42

CLI Estimation Commands

# Set story points
$ dt ticket update 42 points 5

# Update with multiple fields
$ dt ticket update 42 points 5 level moderate
