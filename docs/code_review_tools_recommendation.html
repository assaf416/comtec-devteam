Code Review & Approval in CI

A comprehensive evaluation of automated code review tools for our multi-language stack — with emphasis on on-premises deployment, AI-assisted reviews, and CI pipeline integration.

📋 Requirements

Based on our team's stack and infrastructure constraints, the ideal code review/approval tool must meet:

Language Coverage

ProjectLanguages / Frameworks

Must-Have Requirements

  • On-premises deployment — All code stays on our internal network. No source code sent to external clouds.
  • Multi-language support — Ruby, C#, TypeScript, JavaScript, VB.NET at minimum.
  • CI integration — Runs as a step in Jenkins pipelines and GitHub Actions.
  • Gitea compatibility — Posts review comments on Gitea pull requests via API.
  • Static analysis — SAST (security), linting, code smells, complexity metrics.
  • PR gate — Can block merge if quality thresholds are not met.

Nice-to-Have

  • AI-powered review suggestions (on-prem LLM or self-hosted AI)
  • Historical quality trends and dashboards
  • Custom rules per project/language
  • IDE integration (VS Code, Visual Studio)
  • Low resource footprint (we run on a single Windows Server)

🏆 Recommendation

SonarQube Community Edition + Ollama AI Review
On-premises, free, covers all our languages, with AI-assisted review for complex changes

Why SonarQube Community Edition

  1. Full language coverage — Ruby, C#, VB.NET, TypeScript, JavaScript, HTML, CSS, SQL — all in one tool.
  2. Free and open source — Community Edition covers all our needs at zero license cost.
  3. On-premises — Docker deployment on our existing Windows Server. No code leaves the network.
  4. Quality Gates — Define pass/fail criteria for coverage, duplications, security hotspots, and code smells. Block merges that don't meet standards.
  5. Jenkins plugin — First-class sonar-scanner integration. Results post back to Gitea PRs.
  6. OWASP Security — Built-in security rules for OWASP Top 10, CWE, and SANS vulnerabilities.
  7. Historical trends — Track quality over time per project. Identify regression early.
  8. IDE integration — SonarLint extension for VS Code and Visual Studio (free).

Why Add Ollama for AI Review

  1. Fully on-premises — Ollama runs local LLMs (DeepSeek Coder V2, Qwen2.5-Coder 32B). No API calls to cloud.
  2. Complements SonarQube — SonarQube catches rule-based issues; the AI reviews logic, naming, architecture, and design patterns.
  3. Free — Open-source models. Only cost is hardware (GPU recommended but CPU works for small diffs).
  4. Customisable prompts — We tailor review prompts per language and project conventions.

Phase 4: Jenkinsfile Integration

Jenkinsfile — Rails Project (DevTeam Hub)

pipeline {
  agent any

  environment {
    SONAR_TOKEN = credentials('sonarqube-token')
    SONAR_HOST  = 'http://sonarqube:9100'
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }
    }

    stage('Install') {
      steps {
        sh 'bundle install --jobs 4'
        sh 'yarn install --frozen-lockfile'
      }
    }

    stage('Lint') {
      steps {
        sh 'bundle exec rubocop -f json -o rubocop-report.json || true'
      }
    }

    stage('Test') {
      steps {
        sh 'bundle exec rails db:test:prepare'
        sh 'COVERAGE=true bundle exec rspec --format documentation'
      }
    }

    stage('Security') {
      parallel {
        stage('Brakeman') { steps { sh 'bundle exec brakeman -q --no-pager' } }
        stage('Bundler Audit') { steps { sh 'bundle exec bundler-audit check --update' } }
      }
    }

    stage('SonarQube Analysis') {
      steps {
        sh """
          docker run --rm \\
            --network host \\
            -e SONAR_HOST_URL=\${SONAR_HOST} \\
            -e SONAR_TOKEN=\${SONAR_TOKEN} \\
            -v \${WORKSPACE}:/usr/src \\
            sonarsource/sonar-scanner-cli \\
            -Dsonar.projectBaseDir=/usr/src
        """
      }
    }

    stage('Quality Gate') {
      steps {
        timeout(time: 5, unit: 'MINUTES') {
          sh """
            # Poll SonarQube Quality Gate status
            RESULT=\$(curl -s -u \${SONAR_TOKEN}: \\
              "\${SONAR_HOST}/api/qualitygates/project_status?projectKey=dev-team-hub" \\
              | grep -o '"status":"[^"]*"' | head -1)
            echo "Quality Gate: \$RESULT"
            echo "\$RESULT" | grep -q '"OK"'
          """
        }
      }
    }
  }

  post {
    always {
      // Callback to DevTeam Hub API
      sh """
        curl -s -X POST \${DEVTEAM_HUB_URL}/api/v1/ci_runs \\
          -H 'Authorization: Bearer \${DEVTEAM_API_TOKEN}' \\
          -H 'Content-Type: application/json' \\
          -d '{"ci_run": {"build_number": "\${BUILD_NUMBER}", "status": "\${currentBuild.result}", "branch": "\${env.BRANCH_NAME}"}}'
      """
    }
  }
}

pipeline {

      steps {
        sh 'dotnet restore'
        sh 'dotnet build --no-restore'

    }

    stage('SonarQube End') {
      steps {
        sh 'dotnet sonarscanner end /d:sonar.token="\${SONAR_TOKEN}"'
      }
    }
  }
}

Phase 6: Ollama AI Review Script

#!/usr/bin/env ruby
# ci/ai_review.rb — On-prem AI code review via Ollama
# Usage: ruby ci/ai_review.rb [base_branch]
#
# Sends the git diff to a local Ollama LLM and posts
# review comments back to Gitea via API.

require "net/http"
require "json"
require "uri"

OLLAMA_URL   = ENV.fetch("OLLAMA_URL",   "http://localhost:11434")
OLLAMA_MODEL = ENV.fetch("OLLAMA_MODEL", "qwen2.5-coder:32b")
GITEA_URL    = ENV.fetch("GITEA_URL",    "http://gitea:3001")
GITEA_TOKEN  = ENV.fetch("GITEA_TOKEN",  "")
MAX_DIFF_LINES = 500  # Skip AI review for huge diffs

base_branch = ARGV[0] || "main"

# 1. Get the diff
diff = `git diff #{base_branch}...HEAD --unified=3 -- '*.rb' '*.cs' '*.ts' '*.js' '*.vb'`
if diff.strip.empty?
  puts "No code changes to review."
  exit 0
end

# Truncate very large diffs
lines = diff.lines
if lines.size > MAX_DIFF_LINES
  diff = lines.first(MAX_DIFF_LINES).join
  diff += "\n... (diff truncated to #{MAX_DIFF_LINES} lines)\n"
end

# 2. Build the prompt
prompt = <<~PROMPT
  You are a senior code reviewer. Review the following git diff and provide
  concise, actionable feedback. Focus on:
  - Bugs and logic errors
  - Security vulnerabilities (OWASP Top 10)
  - Performance issues
  - Code style and readability
  - Missing error handling or edge cases

  Format your response as a numbered list of findings. For each finding:
  - State the file and approximate line
  - Describe the issue
  - Suggest a fix

  If the code looks good, say "LGTM — no issues found."

  ```diff
  #{diff}
  ```
PROMPT

# 3. Call Ollama
uri = URI("#{OLLAMA_URL}/api/generate")
body = { model: OLLAMA_MODEL, prompt: prompt, stream: false }
http = Net::HTTP.new(uri.host, uri.port)
http.read_timeout = 300  # LLMs can be slow on CPU

req = Net::HTTP::Post.new(uri.path, "Content-Type" => "application/json")
req.body = JSON.generate(body)

puts "Sending diff to #{OLLAMA_MODEL} for review..."
res = http.request(req)

unless res.is_a?(Net::HTTPSuccess)
  warn "Ollama error: #{res.code} #{res.body}"
  exit 1
end

review = JSON.parse(res.body)["response"]
puts "\n=== AI Code Review ===\n\n#{review}\n"

# 4. Post to Gitea PR (if in CI with PR context)
pr_number = ENV["GITEA_PR_NUMBER"] || ENV["CI_PULL_REQUEST"]
repo      = ENV["GITEA_REPO"]       # e.g. "devteam/dev-team-hub"

if pr_number && repo && !GITEA_TOKEN.empty?
  comment_uri = URI("#{GITEA_URL}/api/v1/repos/#{repo}/issues/#{pr_number}/comments")
  comment_body = "## 🤖 AI Code Review\n\n#{review}\n\n---\n*Reviewed by #{OLLAMA_MODEL} (on-prem)*"

  comment_req = Net::HTTP::Post.new(comment_uri.path, {
    "Content-Type"  => "application/json",
    "Authorization" => "token #{GITEA_TOKEN}"
  })
  comment_req.body = JSON.generate({ body: comment_body })

  comment_http = Net::HTTP.new(comment_uri.host, comment_uri.port)
  comment_res = comment_http.request(comment_req)

  if comment_res.is_a?(Net::HTTPSuccess)
    puts "Review posted to Gitea PR ##{pr_number}"
  else
    warn "Failed to post to Gitea: #{comment_res.code}"
  end
else
  puts "(No PR context — review printed to stdout only)"
end
