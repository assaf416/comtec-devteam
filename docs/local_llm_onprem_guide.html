🔒 Why a local LLM for on-prem projects

For teams that cannot or will not send their work to a third-party SaaS — for data-residency, NDA, air-gapped or cost reasons — a local model is the only way to get AI assistance while keeping everything on infrastructure you control.

On-prem painHow a local LLM solves it
Can't send source code to OpenAI / AnthropicThe model runs on your own machine — code, tickets and customer data never leave the LAN.
Air-gapped / regulated environmentNo outbound internet dependency at inference time.
Per-seat SaaS AI licensingOne machine serves the whole team at a flat, one-time hardware cost.
Vendor outages & rate limitsA pinned local model is always available and deterministic for CI.
Data residency / customer NDAsEvery prompt and response stays inside your perimeter.
Context: DevTeam Hub already speaks to a local model over a plain HTTP REST API (Ai::OllamaClient → Ollama). This guide is about which model and which box to put behind that endpoint.

⚙️ The development features we want to power

The model has to be good at real engineering work, not just chat. These are the jobs DevTeam Hub asks of it today (and where we're heading):

🔍 Code review
Multi-language diff review (Go / Ruby / C# / Node)
  • Finds bugs, security & lint issues, suggests fixes
  • Needs strong code reasoning + long context for big diffs
🐛 Fix errors from tickets
Read a bug ticket → diagnose root cause → propose a fix
  • Benefits from agentic coding skill (read files, edit, run tests)
  • Devstral / Qwen3-Coder shine here
🧩 Break stories into estimated tasks
Decompose a story, calibrate estimates on history
  • Needs reasoning + instruction-following + structured output
📄 Generate documents & specs
Specifications from user stories, runbooks, READMEs
  • Rewards strong prose + structure (Gemma, Qwen, Phi)
📊 Generate presentations of current code/status
Summarise a repo or project state into slides
  • Summarisation + long context over code & metrics
🧪 Test review & generation
Critique Cucumber/specs, suggest missing scenarios
  • Code + test reasoning; medium context
Takeaway: a single code-specialised model (Qwen3-Coder) covers most of this; pair it with a small embedding model for search/RAG and optionally a prose-leaning model (Gemma) for documents and slides.

🖥️ Hardware — which Mac mini M5 to buy

Apple Silicon is an excellent fit for local LLMs: the unified memory is shared between CPU and GPU, so the model weights sit directly in fast memory and Ollama uses the Metal GPU backend automatically. On a Mac mini, RAM is the single most important spec — it caps the model size and context window you can run.

It runs the recommended Qwen3-Coder 30B-A3B workhorse comfortably with room for a large context window and a concurrent embedding model — covering every DevTeam Hub feature. Rule of thumb: keep 30–40% of RAM free for the KV-cache (context) and the OS. A 32 GB box realistically dedicates ~20 GB to weights — exactly a 30B model at Q4.

📐 Memory & quantization cheat-sheet

Quantization shrinks the weights to fit memory. Q4_K_M is the standard sweet spot — ~4 bits per weight, near-full quality. The quick estimate:

RAM for weights ≈ params(billions) × 0.6 GB        (at Q4_K_M)
total RAM needed ≈ weights + 2–6 GB for context (KV-cache) + OS

examples:
  7B   → ~5 GB    (runs on 16 GB easily)
  14B  → ~9 GB    (16 GB ok)
  24B  → ~15 GB   (24 GB comfortable)
  30B  → ~19 GB   (32 GB sweet spot; 24 GB tight)
Your RAMRun comfortablyLargest sensible model
16 GB7B–14B at Q4Qwen2.5-Coder 14B
24 GBup to ~24B at Q4Devstral 24B / Gemma 3 27B (tight)
32 GB ⭐30B MoE at Q4 + context + embeddingsQwen3-Coder 30B-A3B

Quality dial: prefer a bigger model at Q4 over a smaller model at Q8. For code, drop below Q4 (e.g. Q3) only as a last resort — accuracy degrades quickly.

🚀 Setup — from box to DevTeam Hub

1 · On the Mac mini

# Install Ollama (https://ollama.com/download), then expose it on the LAN
launchctl setenv OLLAMA_HOST "0.0.0.0:11434"
ollama serve

# Pull the workhorse (and optional companions)
ollama pull qwen3-coder:30b-a3b-q4_K_M
ollama pull gemma3:27b           # optional — nicer docs/slides
ollama pull bge-m3               # optional — embeddings for search/RAG

# Smoke test
curl http://localhost:11434/api/tags

2 · Point DevTeam Hub at it

Set the environment variables (see .env.example) so Ai::OllamaClient reaches the Mac mini:

OLLAMA_URL=http://your-mac-mini.local:11434
OLLAMA_MODEL=qwen3-coder:30b-a3b-q4_K_M
OLLAMA_TIMEOUT=300

That's it — the AI Agent panel (ticket readiness, code review, fix-that-bug, task breakdown), the live sprint analysis, and the document generators (status presentation, specification) all run against this model. The AI Reports page shows the connection status and every run as an auditable AiReview.

First call is slow. The model loads into memory on first use; subsequent calls are fast. Raise OLLAMA_TIMEOUT and consider an ollama run <model> "" warm-up at boot.
Networking: keep the Mac mini on a trusted VLAN; Ollama has no auth, so restrict access to the DevTeam Hub host (firewall / reverse proxy) rather than exposing port 11434 broadly.
