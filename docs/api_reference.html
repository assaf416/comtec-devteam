REST API Reference For Client Apps, CLI, And The VS Code Extension

This document describes every currently supported endpoint under /api/v1 in the app. It is based on the live Rails routes and controller behavior, including bearer-token authentication, request payloads, curl examples, query filters, and representative JSON response shapes.

Request And Response Format

Base URL

http://localhost:5000/api/v1

All examples below assume the local Rails server is running on port 5000.

Authorization Header

Authorization: Bearer YOUR_API_TOKEN

API requests resolve the token against User.api_token. No session cookie is required.

Unauthorized requests return 401 with { "error": "Unauthorized" }.

Authentication

All API requests are stateless and require a bearer token.

Request Convention

Send the token in the Authorization header. Requests and responses are JSON. The API disables CSRF session protection for this namespace and authenticates directly against the token.

Common headers
Authorization: Bearer YOUR_API_TOKEN
Content-Type: application/json
Accept: application/json

Error Behavior

If the token is missing or invalid, the API returns HTTP 401.

401 response
{
  "error": "Unauthorized"
}

Checkout Endpoint

Branch bootstrap metadata for CLI and editor integrations.
POST
Auth

Canonical Ticket Checkout

/api/v1/checkout

Returns the branch name, repo URL, base branch, and SSH clone URL for a ticket. If the ticket is unassigned or the request includes assign=true, the ticket is assigned to the current user and set to in_progress.

curl
curl -s -X POST \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  http://localhost:5000/api/v1/checkout \
  -d '{
    "ticket_id": 88,
    "assign": true
  }'
response
{
  "branch": "feature/T-88-add-deployment-widgets",
  "ticket_id": 88,
  "title": "Add deployment widgets",
  "repo_url": "http://gitea.local/devteam/dev-team",
  "base": "main",
  "clone_ssh": "git@gitea.local:devteam/dev-team.git"
}

Create Ticket

/api/v1/tickets

Creates a ticket. The owner defaults to the current API user unless owner_id is provided.

Permitted body fields: project_id, title, description, status, priority, kind, level, how_to_reproduce, assignee_id, owner_id, dev_estimate_hours, tester_estimate_hours.

curl
curl -s -X POST \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  http://localhost:5000/api/v1/tickets \
  -d '{
    "ticket": {
      "project_id": 2,
      "title": "Printer timeout on floor 2",
      "description": "Opened from the desktop app",
      "status": "open",
      "priority": "high",
      "kind": "bug_fix"
    }
  }'

Create Test Result

/api/v1/ci_runs/:ci_run_id/test_results

Stores a suite result for the run. On success the parent CI run is automatically updated to passed when failed == 0, otherwise failed. If the run has no finished_at, the endpoint stamps it.

Permitted fields: suite_name, total, passed, failed, skipped, duration_ms, xml_report.

curl
curl -s -X POST \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  http://localhost:5000/api/v1/ci_runs/15/test_results \
  -d '{
    "test_result": {
      "suite_name": "Playwright smoke",
      "total": 12,
      "passed": 11,
      "failed": 1,
      "skipped": 0,
      "duration_ms": 18900,
      "xml_report": "<testsuite></testsuite>"
    }
  }'
response

  "total": 12,

Suggested Client Flows

Reference sequences for the extension and other client apps.

VS Code Extension

Typical flow:

1. POST /tickets to create work from the editor.
2. POST /checkout to get branch metadata.
3. POST /pull_requests when code is review-ready.
4. POST /ci_runs to trigger Jenkins validation.
5. GET /ci_runs/:id and GET /ci_runs/:id/test_results to show feedback.
6. POST /deployments and GET /deployments/:id for release workflows.

Client Apps And Automation Bridges

Typical flow:

1. Discover memberships with GET /me and GET /projects.
2. Create or update tickets as incidents arrive.
3. Trigger CI runs or submit test results from external runners.
4. Create deployment records and poll deployment status during rollout.
5. Rotate tokens with POST /token/regenerate when needed.
